# План разработки frontend-макета

## 1. Цель проекта

Сверстать полноценный frontend-шаблон SaaS-интерфейса **"Реактивация клиентов для Bitrix24"** по 5 предоставленным PNG-макетам:

1. Dashboard
2. Рекомендации
3. Клиенты в зоне риска
4. Купоны / Рекомендации
5. Настройки системы

Результат должен быть не статичной картинкой, а почти готовым frontend-шаблоном продукта: с общими компонентами, mock data в JS, интерактивными элементами, графиками, таблицами и адаптивом.

## 2. Ключевые решения перед разработкой

### Формат проекта

Делаем **5 отдельных HTML-страниц** + общие CSS/JS-файлы.

Плановая структура:

```text
index.html              // Dashboard
recommendations.html    // Рекомендации
risk-clients.html       // Клиенты в зоне риска
coupons.html            // Купоны / Рекомендации
settings.html           // Настройки системы

css/
  style.css

js/
  data.js
  app.js

assets/
  icons/
  img/
```

Такой формат ближе к ТЗ, удобнее для сдачи клиенту и проще для будущей backend-интеграции.

### Технологии

- HTML + CSS + JS
- JS предпочтительно на jQuery
- ApexCharts для графиков
- SVG-иконки, без emoji
- Bootstrap и тяжёлые UI-фреймворки не использовать
- Без inline-стилей
- CSS variables использовать обязательно
- Mock data хранить в `js/data.js`
- Общую интерактивность хранить в `js/app.js`

### Базовый масштаб

Базовая контрольная ширина макетов: **1158px**.

Правила:

- на `1158px` интерфейс должен выглядеть максимально близко к PNG, практически 1 в 1;
- меньше `1158px` - блоки сужаются и адаптируются;
- mobile - полноценный responsive layout;
- больше `1158px` - адаптация в сторону full screen, без ощущения маленького фиксированного контейнера.

Важно: фактические PNG экспортированы в других размерах (`1536x1024` для 4 страниц и `1024x1536` для настроек), но рабочим ориентиром считаем именно клиентское уточнение про `1158px`.

## 3. Общая разработческая стратегия

Разработка идёт не сразу "страница за страницей с нуля", а через общую систему компонентов.

Порядок:

1. Подготовить структуру проекта.
2. Собрать общий UI-kit и CSS-переменные.
3. Собрать общий layout: sidebar, topbar, main area, help-блок.
4. Реализовать страницы по очереди.
5. Подключить mock data и графики.
6. Довести интерактивность.
7. Сделать адаптив.
8. Провести финальную проверку.

Это снижает риск дублей, расхождения стилей и проблем с адаптивом.

## 15. Основные риски и как их закрываем

### Риск 1: PNG экспортированы не в 1158px

Решение:

- ориентируемся на клиентское уточнение: `1158px` - контрольная ширина;
- на `1158px` делаем максимально близкое совпадение;
- фактический размер PNG используем как визуальный референс, но не как CSS-ширину.

### Риск 2: sidebar отличается между страницами

Решение:

- делаем единый sidebar с полным набором пунктов;
- активный пункт меняется по странице;
- визуал активного состояния повторяет макеты.

### Риск 3: таблицы широкие для mobile

Решение:

- используем горизонтальный скролл внутри table wrapper;
- не допускаем горизонтального скролла всей страницы;
- сохраняем читаемость колонок.

### Риск 4: много интерактивных элементов

Решение:

- сначала реализуем общий механизм dropdown/action-menu/tooltip;
- потом применяем его на страницах;
- mock-поведение делаем достаточным для демонстрации.

### Риск 5: график funnel/pyramid

Решение:

- line, donut и bar делаем через ApexCharts;
- funnel/pyramid imitation делаем кастомным CSS/SVG-блоком, чтобы точнее попасть в макет.

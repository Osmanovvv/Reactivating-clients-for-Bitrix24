# Итоговое ТЗ

## Проект

Сверстать полноценный frontend-макет SaaS-интерфейса **"Реактивация клиентов для Bitrix24"** по предоставленным `.png`-макетам.

Нужно реализовать 5 экранов:

- Dashboard
- Рекомендации
- Клиенты в зоне риска
- Купоны
- Настройки

## Базовый масштаб макетов

Базовая ширина макетов: **1158px**.

Требования к масштабированию:

- на ширине **1158px** интерфейс должен выглядеть максимально близко к макету, практически 1 в 1;
- на ширинах меньше 1158px блоки должны сужаться и адаптироваться под доступное пространство;
- на мобильных устройствах должна быть полноценная адаптация интерфейса;
- на ширинах больше 1158px интерфейс должен адаптироваться в сторону full screen, сохраняя общий premium/SaaS-визуал, пропорции, spacing и читаемость.

## Технологии

- HTML + CSS + JS
- JS предпочтительно на jQuery
- Все данные статические, mock data хранится внутри JS
- Вёрстка должна быть максимально готовой к дальнейшей интеграции с backend
- После завершения frontend-работ должно остаться только подключить backend и заменить mock data на API, без переделки дизайна/UI

## Структура проекта

Требуется чистая и понятная структура проекта:

- отдельные файлы для CSS и JS;
- без inline-стилей;
- нормальные, читаемые классы;
- допускается BEM-like naming;
- допускаются CSS variables;
- допускается SCSS, если это удобно;
- Bootstrap и другие тяжёлые UI framework не использовать.

## Общие UI-требования

Интерфейс должен быть не статичной картинкой, а живым frontend-шаблоном продукта.

Нужно сохранить:

- светлый минималистичный SaaS-стиль;
- внешний вид максимально близкий к макетам;
- скругления;
- тени;
- spacing;
- hover-эффекты;
- premium-визуал;
- аккуратные интерактивные состояния элементов.

Цветовая схема:

- фиолетовый primary;
- зелёный success;
- жёлтый warning;
- красный danger.

## Интерактивность

Все элементы интерфейса должны быть интерактивными.

Необходимо реализовать:

- hover-состояния;
- dropdown/select;
- кнопки;
- переключатели;
- пагинацию;
- tooltip/help icons;
- модальные или раскрывающиеся элементы, если они предусмотрены макетами;
- фильтры;
- search input;
- action buttons;
- badges/status labels;
- hover-анимации;
- минимальные empty/loading states.

## Основные компоненты

Должны быть реализованы:

- sidebar navigation;
- header/topbar;
- KPI-карточки;
- таблицы;
- фильтры;
- dropdown;
- search input;
- charts;
- progress bars;
- badges/status labels;
- action buttons;
- tooltip/help icons;
- pagination;
- responsive behavior;
- hover animations;
- empty/loading state, минимально;
- sticky header/sidebar, по возможности.

## Графики и диаграммы

Использовать популярную JS-библиотеку для графиков:

- ApexCharts;
- Chart.js;
- ECharts;
- либо другую по согласованию.

Рекомендуемый вариант: **ApexCharts**, так как он хорошо подходит для SaaS-дашбордов и аккуратно закрывает line, donut и bar chart.

Все графики должны инициализироваться через JS статичными mock-данными.

Нужно реализовать:

- line chart;
- donut chart;
- bar chart;
- funnel/pyramid imitation;
- progress bars.

## Dropdown / фильтры

Если список значений не показан в макете, всё равно нужно реализовать dropdown с примерными данными.

Примерные данные для менеджеров:

- Иван Петров
- Мария Сидорова
- Алексей Кузнецов
- Екатерина Волкова

Примерные статусы:

- Создан
- Отправлен
- Использован
- Просрочен

Примерные значения риска:

- Низкий
- Средний
- Критический

Примерные сегменты:

- VIP
- Retail
- B2B
- Новые клиенты

Примерные периоды:

- 7 дней
- 30 дней
- 60 дней
- 90 дней

## Иконки

Требуются SVG-иконки.

Можно использовать:

- Font Awesome
- Lucide
- Heroicons
- Flaticon SVG

Требования:

- иконки должны быть максимально похожи на макет;
- цвета и стиль иконок должны совпадать с дизайном;
- emoji не использовать.

## Адаптив

Нужна полная адаптация под мобильные устройства.

Sidebar:

- desktop: полный sidebar;
- tablet/mobile: sidebar сужается до иконок;
- mobile: sidebar может скрываться через burger menu.

Таблицы:

- допускается горизонтальный скролл;
- либо карточный responsive layout на усмотрение верстальщика;
- важно сохранить удобство чтения и общий premium/SaaS-визуал.

## Ожидаемый результат

На выходе должен получиться почти готовый frontend-шаблон SaaS-продукта, а не демо-картинка.

Итоговая вёрстка должна позволять:

- заменить mock data на данные из API;
- подключить backend;
- сохранить текущий дизайн и UI без переделки;
- использовать интерфейс как основу реального продукта.
